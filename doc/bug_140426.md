## 2026-04-14 — Ingest hangs silently before pipeline starts

**Files changed:** `src/downloader.py`, `main.py`

### Symptom
Running `ingest` with a YouTube URL caused the CLI to freeze immediately — before any
progress spinner appeared — requiring a manual Ctrl+C. No output was shown to the user.

### Root cause
Two independent problems in `_cmd_ingest` / `fetch_playlist_entries`:

1. **No socket timeout on yt-dlp** — neither `fetch_playlist_entries()` nor `download_audio()`
   set `socket_timeout` in their `ydl_opts`. yt-dlp's underlying `requests` session used the
   default OS socket timeout (effectively infinite), so a slow or unresponsive server blocked
   the process forever.

2. **No UI feedback before `fetch_playlist_entries()`** — the call at `main.py:99` was bare:
   ```python
   entries = fetch_playlist_entries(args.url)
   ```
   No spinner, no status message. The user saw a blank terminal and had no way to know
   whether the app was working or frozen.

Note: the `download_audio()` function has the same timeout gap, but it runs inside the
pipeline where `_ProgressHandler` already shows a spinner, so the hang is less alarming.

### Fix
**`src/downloader.py`** — added `"socket_timeout": 15` to `ydl_opts` in both
`fetch_playlist_entries()` and `download_audio()`. yt-dlp propagates this to the underlying
`urllib3` connection, so any stalled network call will raise after 15 seconds instead of
waiting forever.

**`main.py`** — wrapped the `fetch_playlist_entries()` call with `ui.console.status()`:
```python
with ui.console.status("[bold cyan]Fetching…[/bold cyan]"):
    entries = fetch_playlist_entries(args.url)
```
This reuses the same Rich spinner already used by `_ProgressHandler`, giving immediate
visual feedback that the app is working.
