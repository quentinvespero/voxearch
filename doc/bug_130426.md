## 2026-04-12 — Transcription step hangs on network even with cached model

**Files changed:** `src/transcriber.py`, `src/pipeline.py`

### Symptom
`ingest` stalled at step 2 ("Transcribing…") indefinitely, requiring a manual Ctrl+C.
Affected every run, including runs where the Whisper model was already in the local cache.

### Root cause
`src/transcriber.py` tried to prevent mlx_whisper's internal network call by setting
`os.environ["HF_HUB_OFFLINE"] = "1"` at runtime. This was a no-op.

`huggingface_hub.constants.HF_HUB_OFFLINE` is a **module-level boolean evaluated once
at import time** — changing the env var after import has no effect on it:
```python
# huggingface_hub/constants.py
HF_HUB_OFFLINE = _is_true(os.environ.get("HF_HUB_OFFLINE") ...)  # frozen at import
```
Because the constant stayed `False`, every call to `mlx_whisper.transcribe()` triggered
`snapshot_download` → `api.repo_info()` (HTTP to huggingface.co), which hung indefinitely
on a slow or unavailable connection.

### Fix
**`src/transcriber.py`** — removed the env-var approach entirely. Instead:
- When the model is already cached: calls `snapshot_download(model, local_files_only=True)`
  which is a purely local operation (no network call) and returns the filesystem path.
- On first run: calls `snapshot_download(model, etag_timeout=30)` to download with a
  30-second connection timeout (fails fast instead of hanging forever).

In both cases the resolved **local path** is passed to `mlx_whisper` as `path_or_hf_repo`.
`mlx_whisper`'s `load_model` checks `Path(path_or_hf_repo).exists()` before calling its
own `snapshot_download`, so the network call inside mlx_whisper is never reached.

**`src/pipeline.py`** — step 2 spinner label is now `"Downloading & transcribing"` on
first run and `"Transcribing"` on subsequent ones, so the user knows what is happening.
